### Major — Dual Organization-Context Architecture (real bug surface)

`App.tsx:355–359` mounts both `OrgProvider` (from `context/OrgContext.tsx`) and `OrganizationProvider` (from `context/OrganizationContext.tsx`) simultaneously:
- `OrgContext` writes to `sessionStorage` using keys `ACTIVE_SCOPE`/`ACTIVE_ORG_ID`.
- `OrganizationContext` writes to `localStorage` using the same logical keys.
- Both dispatch `orgScopeChanged` window events.
- The vestigial `OrgContext` path's four consumers (`OrgSwitcher`, `AddKeywordModal`, `AddAgentModal`, `AddMemoryBlockModal`) are all dead (none mounted in `App.tsx`).

Result: any future code that reads from one storage backend and writes to the other will introduce silent divergence. This is not a theoretical concern — it is a write-conflict waiting to be triggered by a new session-restore feature.

File: `apps/hindsight-dashboard/src/App.tsx:355–359`; `apps/hindsight-dashboard/src/context/OrgContext.tsx`; `apps/hindsight-dashboard/src/context/OrganizationContext.tsx`.

**How to detect regression:** Delete `context/OrgContext.tsx`, `api/orgsService.ts`, and their four dead consumers in a single PR. Then verify `App.tsx` has exactly one organization provider.

## Summary

**Drift score: 47/100 (Stabilize band)**. The quantitative backbone is dominated by layer-boundary violations: 8 mutating backend routes with no auth dependency, a shadowed superadmin handler that is permanently unreachable, 3 TypeScript API modules importing the browser toast bus, and 3 search endpoints duplicating auth logic while skipping PAT scope narrowing. The file-size signal is in the Watch band (8.7%), not Critical — but three files exceed 1000 LOC (`main.py` 1353, `notification_service.py` 1277, `search_service.py` 1191) and represent concentrated technical debt.

**Top 3 doc-vs-code drifts:**
1. `docs/architecture.md` is auto-generated and was not re-run after 14 Python modules were added (beta access, PAT tokens, embedding, query expansion, feature flags, runtime, token crypto) — these functional areas are architecturally invisible in the only module-level index that exists.
2. `core/db/models/users.py:15` model comment enumerates 4 beta-access states; the live code has 5 (`revoked` is reachable and documented in `authentication_flow.md`).
3. `docs/architecture.md` entirely omits TypeScript module documentation — 97 source files with zero architectural description in any generated doc.

**Top 3 structural drifts:**
1. Eight mutating routes (`/memory/prune/*`, `/memory-blocks/*compress*`, `/memory-blocks/bulk-*`, `/consolidation/trigger/`, `/consolidation-suggestions/{id}/delete`) lack auth dependencies — the global middleware is the only guard, and it does not validate identity for these routes.
2. `core/api/main.py` (1353 LOC) functions simultaneously as an app factory, a set of 17 inline endpoint handlers, and an embedded 130-line text-processing utility — three responsibilities in one file that is already the single largest source file in the backend.
3. Dual organization-context architecture (`OrgContext` + `OrganizationContext` both mounted in `App.tsx`) writes to two different storage backends for the same logical keys and dispatches the same window event — a dormant write-conflict that will manifest when any future feature reads from one store and writes to the other.

**One prevention measure**: Add `pre-push` hooks that (a) run `python scripts/generate_architecture_docs.py && git diff --exit-code docs/architecture.md` to catch stale docs on every push, and (b) run a pytest fixture that asserts every registered `POST`/`PUT`/`PATCH`/`DELETE` FastAPI route has a declared auth dependency — catching unauthenticated mutating endpoints before they reach staging.
